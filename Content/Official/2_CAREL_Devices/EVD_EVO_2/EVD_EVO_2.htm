General operations

EVD_EVO_2 manages communication between a c.pCO and the EVD driver to control the electronic expansion valve, leaving the driver the task of managing all the control processes.
For the Function Block to work, all pins in the shape must be connected.

When EVD evolution is connected to a pCO, the displayed settings are read only (write disabled).

Upload on target: whenever uploading a new application with EVD_EVO_2 on the devices, or when wanting to restore the unit configuration, it is strongly recommended to clear the permanent memory.
In order to clear the memory, open the system screen by pressing “Alarm + Enter” for a few seconds; browse to Application > Wipe Retain and press Enter.

EVD evolution and pCO synchronisation

The c.pco – EVD evolution system is a master-slave network where the c.pco acts as the master and EVD evolution the slave: in this configuration, the EVD evolution parameters can only be set by c.pco using the variables and screens provided by the function block, or by running the default installation procedure.

When EVD evolution is connected to a pCO, the displayed settings are read only (write disabled).

Each time the c.pco – EVD evolution system is turned on, the master (pCO) automatically forces synchronisation with EVD evolution. All the values in the c.pco memory are sent to EVD evolution and the output pin DrvReadyToGo_EVD = TRUE signals that synchronisation has been completed. The output variable DrvReadyToGo_EVD is set when all the following conditions are met:

  1. After boot or failed EVD evolution connection when valves are in Standby. This indicates that the driver has forced closing and is ready for control .

    For twin drivers, both valves must be in standby.

  2. After transferring data from c.pco to EVD evolution. This means that the c.pco and driver are aligned;
  3. When there are no failed connection alarms between c.pco and EVD evolution;
  4. When the default installation procedure is not running

Output variable DrvReadyToGo_EVD thus signals when a VlvA_EVD.CoolCap/ VlvB_EVD.CoolCap > 0 can be sent to the driver. In fact, the compressor should only start when the EVD evolution is ready to control valve opening.

The driver only accepts circuit power from pCO if the driver's digital input settings are NOT ”Start/Stop" control. Alternatively, use the driver's digital inputs.

The following paragraph shows how to set the EVD with:

AC or chiller with BLDC scroll compressor

Regulation 19 "AC or chiller with BLDC scroll compressor" is used for BLDC compressors to keep discharge temperature under control.

EVD_EVO_2 receives data from BLDC through input pins "BLDC_ParamsIn_EVD", "RegSubTyp_EVD" and "S4_NTC_ThT_K_EVD"

  • BLDC_ParamsIn_EVD contains the following parameters:
    • Dscg_SHSet, discharge superheat set point (RegSubTyp_EVD = 2)
    • Dscg_TempSet, discharge temperature set point (RegSubTyp_EVD = 3)
    • MOP_ThrshFastChg, MOP threshold for fast change
  • RegSubTyp_EVD defines the current set point:
    1Superheat set point
    2Discharge superheat set point
    3Discharge temperature set point
  • S4_NTC_ThT_K_EVD is a Thermal time constant for NTC probe S4

The discharge conditions can be read via output variables "BLDC_Meas_EVD.Dscg_SH" and "BLDC_Meas_EVD.Dscg_Temp".

This control function is available only for CAREL single valve drivers. No set point needs to be configured by the user; they are configured by "Comp_BLDC" function block.

For more information see "Comp_BLDC" documentation, paragraph "Example" and EVD driver manual, paragraph "BLDC Control with compressor"

Modulating MOP threshold (BLDC compressors)

This function is used with BLDC compressors to return back inside the allowed envelope. The MOP threshold is set by the input variable BLDC_ParamsIn_EVD.MOP_ThrshFastChg: this information typically comes from the BLDC function block.

The function is activated only if main control is set as control for BLDC compressors (Main control 19).

For more information see "Comp_BLDC" documentation, paragraph "Example" and EVD driver manual, paragraph "BLDC Control with compressor"

4. Other EVD configurations

 

Operations in fieldbus serial mode

Cooling capacity and, consequently EVD evolution start/stop, is sent via network via "VlvA_EVD_1.CoolCap" and "VlvB_EVD_1.CoolCap", but only if EVD evolution digital inputs are not set as ''Start/Stop control''. If one of the two digital inputs is set as control backup (I85_DIGIN1_CONFIG_msk = 6 or 7), the driver will only follow the input status in the event of offline.

For further information, see the digital input priority paragraph in the driver manual +0300006 on www.carel.com.

 Evd Evolution working in DC direct current

To run EVD evolution in direct current (24VDC), set the input variable CfgParam_EVD.EN_DC_Supply. In this case, in the event of a power outage, there is no forced emergency valve closing on EVD Evolution.

 

 Built-in Evd Evolution control on c.pCO

The c.cpo medium size board can have a Built-in EVD evolution driver. From a control standpoint, this can be considered as a remote driver connected to Fieldbus 2. The communication protocol to be used is Modbus Master.

The driver always responds to network address 198 and cannot be changed. The firmware version must be >5.6 to manage the Built-in EVD evolution driver with EVD_EVO_2 function block.

The driver is powered in direct or alternating current based on the type of c.pco power. Parameter CfgParam_EVD.EN_DC_Supply must be correctly set in order to ensure communication.

 

 Work As Single mode

A TWIN driver can be set as SINGLE with EVD_EVO_2 by setting pin WorkAsSingle_EVD to TRUE. If this function is enabled, the driver acts as a SINGLE model, permitting, for example, the use of auxiliary control functions.

these settings are visually signalled on the driver by the two OPEN and CLOSE LEDs blinking for the second valve.

if the connected driver is SINGLE, setting pin WorkAsSingle_EVD to TRUE has no effect.

 

 Frequently changed parameters

To disable parameter saving to permanent memory and thus allowing parameters to be frequently changed, uncheck the “Retain” flag for the following variables on inputs "VlvAIn_EVD" and "VlvBIn_EVD":

• SH_SET

• PID_Kp

• PID_Ti

• PID_Td

• LOP_Threshold

• LOP_Ti

• MOP_Threshold

• MOP_Ti

• LOW_SH

• LOW_SH_Ti

EVD_EVO_2 set EVD evolution driver to save parameters in its buffer memory (EEPROM) every two hours. This is to protect the device's EEPROM memory.

5. EVD evolution default value installation procedure

Default installation from c.strategy

Using this procedure, users need to set their own default values by assigning a set of constants to the Super, child and normal parameters of the input variables of EVD_EVO_2.

For more information see "Demo_EVD_EVO_2", which contains a template for the default configuration in c.strategy.

Super parameter installation must be completed before installing the child and normal parameters: see "Example 1: default installation from application".

To start the installation, set the input variable "InstDefIn_EVD" = TRUE.

User default values are installed via two output pins connected to super parameter variables (DefSuperParams_EVD) and child and normal parameter variables (DefParams_EVD) respectively.

This procedure is run with the FB enabled, but not necessarily with the driver online. The module will automatically program the driver with the loaded values once communication has been established.

It is possible to require the default installation with the block disabled: the request will be saved in the "Default_MemoryIn_EVD" input, which must be retain. For more information see "Demo_EVD_EVO_2".

EVD_EVO_2 function block must never be bypassed by a Jump during the installation procedure.
The default installation procedure can be queued after normal application defaults without having to reboot the controller twice.

 
Default installation from DEV

Default values can also be set in the DEV editor in c.design. To perform a default installation, first run a Wipe Retain from the System menu: open the system screen by pressing “Alarm + Enter” for a few seconds, browse to Application > Wipe Retain and press Enter. For more information, download Demo_EVD_EVO_2 from KSA.

For both default procedures, see Example 1 and 2 at the end of the documentation.

 

 

6. Communication protocol settings

Modbus Master protocol is used for communication with the c.pco.

The protocol type is the same for all application drivers. Set the driver network address with pin DevAddr_EVD in every EVD_EVO_2 FB.

See the EVD Evolution manual for the relevant communication settings.

Modbus Master

For communications via Modbus Master protocol, import the default templates.

EVD_EVO_2 requires six templates:

EVD_EVO_2_Mng: manages basic parameters to establish communication
EVD_EVO_2_ParamsA: manages valve A control parameters
EVD_EVO_2_ParamsB: manages valve B control parameters
EVD_EVO_2_EEVA: manages valve A parameters
EVD_EVO_2_EEVB: manages valve B parameters
EVD_EVO_2_PrbAlrmDT: manages Probe alarm delays (*)

* Available from library version 1.4.0 and EVD FW >= 6.6. The function block manages the enable of the template accordingly to the EVD FW version (backward comparability with FW < 6.6)

Managing the unit of measure

EVD_EVO_2 manages only the International System set of units of measure (°C/barg).
Once the EVD driver has been connected to the c.pCO, this sets the UoM of the driver to International System units, without the possibility of modification.

In EVD_EVO_2, the units of measure are managed in the ST code of c.mask and can only be changed for displaying the screens: for more information on how to manage the UoMs, see Demo_EVD_EVO_2 on KSA.

• Data transmission errors from pCO to EVD evolution.

7.1 Firmware compatibility

If the connected EVD evolution FW version is lower than 5.6, the FB cannot be run: consequently, communication cannot be established with c.pco and the driver cannot be managed.

If this occurs, the module will warn the user of the error via parameter SysAlrms_EVD (bitfield variable, see table EVD_EVO_2: outputs).

7.2 Data transmission errors from c.pco to EVD evolution.

EVD_EVO_2 incorporates a communication debug system using dedicated variables that indicate when errors occur during data transmission.

The configuration error alarm (bit 15 of output "SysAlrms_EVD") indicates the presence of different kinds of errors:

Super parameter errors

Local variable SuperParams_Configuration_Error signals an error when:

  • The following control functions are set on the TWIN driver:

    • SUPER_MAIN_REGULATION = 19 or 20

    SUPER_AUX_REGULATION = 2, 3, 4, 5, 6, 8, 9
    SUPER_MAIN_REGULATION_2ND = 19 or 20
  • The following valve types are set on the CAREL VALVE driver ONLY :
    • SUPER_EEV_TYPE ≠ 1 or 19
    SUPER_EEV_TYPE_2ND ≠ 1 or 19
  • Control is set on a UNIVERSAL driver:
    • (SUPER_MAIN_REGULATION = 19)
  • Auxiliary control functions set are not coherent for the following main control values:
    • SUPER_MAIN_REGULATION = 19
    SUPER_MAIN_REGULATION = 20
    SUPER_MAIN_REGULATION = 21
    SUPER_MAIN_REGULATION = 22 (*see table Super_Main_Regulation/Super_Aux_Regulation table:)

 

If one of the previous conditions is met, data may bounce back between pCO and EVD evolution or communication could jam on the bus due to continuous pCO write attempts based on the type of communication protocol selected for the network. Parameter sending and reception is generally no longer guaranteed.

 

*Super_Main_Regulation/Super_Aux_Regulation table:

Main control Super Aux control Super
19: AC/Chiller with scroll BLDC compressor 5: AC/Chiller with scroll compressor SIAM ANB
20: SH control with two temperature probes 6: SH control with two temperature probes
21: I/O expansions for pCO 7: I/O expansions for pCO
22: Programmable SH control 10: Programmable SH control

 

Parameter configuration error

EVD_EVO_2 contains 3 primary parameter structures: AN_PARAM, INT_PARAM and DIG_PARAM. Each input parameter of EVD_EVO_2 is assigned to its corresponding variable inside these 3 structures.
Each of these variables is associated with four parameters:

.Val variable value
.MskLimMax maximum value that the variable can take on the user interface
.MskLimMin minimum value that the variable can take on the user interface
.Err value set by the variable integrity control

.Err represents:

0 no error
1 parameter value over maximum limit
2 parameter value below minimum limit

 

 

Both errors can be debugged following the guide below:

How to debug a Configuration error

  1. Upload the software to the c.pCO
  2. Open c.strategy and start the Debug on Target
  3. Add to spylist the instance of EVD_EVO_2
  4. Open the structure "EVD_EVO_2_InstanceNumber.VarErrorDetection_1"
  5. The error can be caused
    • by incorrect settings of a Super Parameter: in this case, the variable "SuperParams_Configuration_Error" in the SpyList will be "TRUE". See "Data transmission errors from c.pco to EVD evolution", paragraph "Super parameters errors"
    • by incorrect parameter configuration: in this case, find the "tmpVal" that gives the error in the spylist. Each "tmpVal" is a set of error values, see the "Error List" chapter in the "EVD_EVO_2_StructuresList" document (on Online help) to find the corresponding set of error values, and then proceed with step 6.
  6. According to the set of error values, in the spylist, open the structure in the EVD_EVO_2 instance "AN_Params" or "INT_Params" or "DIG_Params" and find the variable that has an error code other than "0"

The error can be caused by an incorrect input value or incorrect transmission data with the EVD driver.
To correct the error, check the input values of the function block and, if necessary, the DEV values in DEV editor, or start a Default installation by input pin "InstDef_EVD".

Examples
Example 1: default installation from application

The installation of EVD driver default values involves loading the default values in sequence, first the super parameters and then the child and normal parameters.
Through pins DefSuperParams_EVD and DefParams_EVD, the function block gives the correct logical sequence to install the default values: the following procedure needs to be implemented outside of the block.

  1. Default installation request for EVD evolution through input pin InstDefIn_EVD.

    EVD_EVO_2 function block must never be bypassed by a Jump during the installation procedure.


  2. EVD_EVO_2 requires the default installation settings DefSuperParams_EVD = TRUE. Outside of the block, the super parameter settings procedure must be set to the required default values
  3. Next, the block sets DefSuperParams_EVD = FALSE and DefParams_EVD = TRUE: outside of the block, the child and normal parameter settings procedure must be set to the required default values

For more information see "Demo_EVD_EVO_2", which contains a template for the default configuration in c.strategy.

 

 Example 2: default installation from DEV
  1. To install the default values using DEV, the values need to be set in the DEV editor


  2. To perform the installation using DEV, open the system screen by pressing “Alarm + Enter” for a few seconds; browse to Application > Wipe Retain and press Enter


 Example 3: adding parameters not managed in EVD_EVO_2

EVD_EVO_2 doesn't manage all EVD driver parameters.
To modify the EVD_EVO_2, the source code needs to be requested from Carel CST csuitesupport@carel.com

To manage a new parameter:

  1. Open EVD_EVO_2 source code
  2. Update the following structures with the new parameter (if not present):
    • "ANALOG_P_EVD" , for an analogue parameter
    • "INTEGER_P_EVD", for an integer parameter
    • "DIGITAL_P_EVD", for a digital parameter
    • "PARAMS_UI_EVD" for screen limits management

    • "MASK_FIELD_EN_EVD" for enabling the mask field

  3. Update the EVD_EVO_2 shape with the new input or output parameter, by adding a new pin or updating the existing structures
  4. For a Read-Only parameter, update the EVD_EVO_2 main block, assigning the new parameter (of structure"ANALOG_P_EVD", "INTEGER_P_EVD" or "DIGITAL_P_EVD") to the new output pin and go to step 6
  5. For a Read/Write parameter update the "Input_Mng" function block, assigning the new input of EVD_EVO_2 to the new parameter structures "ANALOG_P_EVD", "INTEGER_P_EVD" and "DIGITAL_P_EVD"
    1. For an ANALOG parameter
      1. Open "RW_AnalogParams_EVD" POU
      2. Add a new "AnalogParams_Mng_1" function block instance, populating the input pins and assigning the output to the parameter structure (see example)

      3. Add the variable .Err in "VarErrorDetection_EVD" function block
    2. For an INTEGER parameter
      1. Open "RW_IntegerParams_EVD" POU
      2. Add a new "IntegerParams_Mng_1" function block instance, populating the input pins and assigning the output to the parameter structure (see example)

      3. Add the variable .Err in "VarErrorDetection_EVD" function block
    3. For a DIGITAL parameter
      1. Open "RW_DigitalParams_EVD" POU
      2. Add a new "DigitalParams_Mng_1" function block instance, populating the input pins and assigning the output to the parameter structure (see example)
  6. Update the "ParamsLimit_Mng_EVD" function block for the screen limits and "CheckMinus7777_EVD" for the parameter field enable
  7. If not present, add the parameter in the Modbus template ("EVD_EVO_2_ParamsA" for valve A parameters and "EVD_EVO_2_ParamsB" for valve B parameters)
