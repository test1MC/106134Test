General operations

For the Function Block to work, all pins in the shape must be connected.

When EVD evolution is connected to a pCO, the displayed settings are read only (write disabled).

For further information on electronic expansion valve driver operations, see manual (EVD evolution or EVD evolution Twin) +0300005 on the www.carel.com website

Operations in fieldbus serial mode

Cooling power and, consequently EVD evolution start/stop, is sent via network through CoolCap_A_EVD and CoolCap_B_EVD, but only if EVD evolution digital inputs are not set as ''Start/Stop regulation''. If one of the two digital inputs is set as regulation Back-up (I85_DIGIN1_CONFIG_msk = 6 or 7), the driver will only follow the input status in the event of offline.

For further information, see the digital input priority paragraph in the driver manual code +0300006 on www.carel.com.

Modulating MOP threshold (BLDC compressors)

This function guarantees BLDC compressor operations with the allowed envelope zone. The MOP threshold the driver uses for protection is communicated by VAR_INPUT BLDC_Params_EVD.MOP_ThrshFastChg.

The function is on only if main regulation is enabled as control for BLDC compressors.

Valve regulation on superheating or discharge temperature (BLDC compressor)

This function lets you monitor BLDC compressor discharge temperature. The reference set point for these regulations is communicated via VAR_OUTPUT BLDC_Measure_EVD.Dscg_SH and BLDC_Measure_EVD.DscgTemp. Regulation is switched by VAR_INPUT RegSubTyp_EVD.

the function is on only if main regulation is enabled as control for BLDC compressors.

Expanded display of pressure probe values

The expanded value (expressed in hundredths of bar or tenths of psi) for probes S1 and S3 can be read using VAR_OUTPUT:

• PrbVal_EVD.S1_ValExtd_EVD for probe S1

• PrbVal_EVD.S3_ValExtd_EVD for probe S3

These values are automatically converted based on the selected unit of measure.

Evd Evolution working in DC direct current

To run EVD evolution in direct current (24VDC), set parameter CfgParam_EVD.EN_DC_Supply. This setting is found as VAR_INPUT. In this case, in the event of power outage, there is no forced emergency valve closing in EVD Evolution.

On board Evd Evolution control on c.pCO

The c.cpo medium size board can have an on board EVD evolution driver. From the control standpoint, this can be considered as a remote driver connected on Fieldbus 2. The communication protocol to be used can be Modbus Master.

The driver always responds to the 198 network address and cannot be changed.

The firmware version must be >5.0 to manage the on board EVD evolution driver with EVD_EVO function block.

The driver is powered in direct or alternate current based on the type of c.pco power. Parameter CfgParam_EVD.EN_DC_Supply must be correctly set in order to communicate.

Super parameters and child parameters

Single child parameters can always be edited. If a child parameter is changed, the super parameter does not change but keeps the previous value (on both EVD evolution and pGD).

EVD evolution and pCO synchronization

The c.pco – EVD evolution system is a master-slave network where the c.pco acts as the master and EVD evolution the slave: in this configuration, EVD evolution parameters can only be set by c.pco using the variables and masks provided by the function block, or by running the default installation procedure.

When EVD evolution is connected to a pCO, the displayed settings are read only (write disabled).

Each time the c.pco – EVD evolution system is turned on, the master (pCO) automatically forces synchronization with EVD evolution. All the values in the c.pco memory are sent to EVD evolution and the output pin DrvReadyToGo_EVD = TRUE signals completed synchronization. The VAR_OUTPUT DrvReadyToGo_EVD is set when all the following conditions are met:

  1. After boot or failed EVD evolution connection when valves are in Stand-by. This indicates that the driver forced closing and is ready for regulation.

    For twin drivers both valves must be in stand-by.

  2. After transferring data from c.pco to EVD evolution. This means that the c.pco and driver are aligned;
  3. When there are no failed connection alarms between c.pco and EVD evolution;
  4. When the default installation procedure is not running

VAR_OUTPUT DrvReadyToGo_EVD thus signals when a VlvA_EVD.CoolCap/ VlvB_EVD.CoolCap > 0 can be sent to the driver meaning when compressor operations can start. In fact, the compressor should only start when the EVD evolution is ready to regulate valve opening.

The driver only accepts circuit power from pCO if the driver's digital input settings are NOT ”Start/Stop regulation. Alternatively, use the driver's digital inputs.

EVD evolution default value installation procedure from Function Block (FB)

The procedure is run by setting input InstDef_EVDIn = TRUE. The procedure loads a default value set by the user in EVD evolution parameters. Values not initialized in this way are automatically set to their default value (see EVD Evolution manual). This procedure is run with the FB enabled, but not necessarily with the driver online. The module will automatically program the driver with the loaded values when the communications are established.

Default values are installed via two output pins connected to super parameter variables (DefSuperParams_EVD) and child and normal parameter values (DefParams_EVD) respectively. See the demo provided at http://ksa.carel.com/it/group/ksa/home  Software & Support  c.suite  Demo & Tutorial.

Pins DefSuperParams_EVD and DefParams_EVD are reset at the end of installation.

EVD_EVO function block must never be bypassed by a Jump during the installation procedure.

The default installation procedure can be queued after normal application defaults without having to reboot the controller twice.

Communication protocol settings

Modbus Master protocols can be used for communications in the c.pco – EVD evolution system with EVD_EVO FB.

The protocol type is unique for all application drivers. Set the driver network address with pin DevAddr_EVD in every EVD_EVO FB.

See EVD Evolution manual for EVD evolution driver models and relevant communication protocols.

Modbus Master

For communications via Modbus Master protocol, import the default templates from c.suite installation folder or contact c.suite customer support at csuitesupport@carel.com.

EVD_EVO FB requires five templates:

  1. EVD_EVO_ModbusMng: manages basic parameters to establish communications
  2. EVDEvolutionModbusParamsA: manages valve A regulation parameters
  3. EVDEvolutionModbusParamsB: manages valve B regulation parameters
  4. EVDEvolutionModbusEEVA: manages valve A parameters
  5. EVDEvolutionModbusEEVB: manages valve B parameters
Managing the unit of measure

EVD evolution manages two sets of units of measure: °C/barg (International System) and °F/psig (Imperial System).

Unit of measure management module architecture is organized so that the same set is used for all application drivers: it is set in Function Block EVD_EVO, VAR_INPUT UnitMeasure_EVD.UnitMeasure. Refer to the relevant documentation.

When c.pco and EVD evolution are online, a unit of measure change request generates the value conversion in the driver which is then sent to pCO.

If the unit of measure is changed with the driver off-line and the driver is then re-connected, EVD evolution will not convert any data: all data from c.pco will be received. The developer must be sure to send data coherent with the selected unit to EVD evolution.

Error warnings

EVD_EVO provides debug variables to warn the user and developer of:

• Firmware compatibility error (SysAlrms_EVD bit );

• Data transmission errors from pCO to EVD evolution.

Firmware compatibility

If the connected EVD evolution FW version is under 5.0, the FB cannot be run: consequently, communications cannot be established with c.pco and the driver cannot be managed.

If this occurs, the module will warn the user of the error via parameter SysAlrms_EVD (bitfield variable, see table EVD_EVO: outputs).

Data transmission errors from c.pco to EVD evolution.

Mod_EVDEVO incorporates a communication debug system using dedicated variables that indicate when errors occur during data transmission.

Each EVD evolution parameter managed in EVD_EVO has a local variable not in shape (for example AN_Params.A50_SH_SET_msk.Err is included for AN_Params.A50_SH_SET_msk) which, based on its value, indicates:

• 0 = no error

• 1 = parameter value over maximum limit

• 2 = parameter value under minimum limit

• 3 = decimal part other than 0 (for psi and psig pressures only)

Local variable variable SuperParams_Configuration_Error signals an error in the event:

• the following regulations are set on TWIN driver:

• SUPER_MAIN_REGULATION = 19 or 20

• SUPER_AUX_REGULATION = 2, 3, 4, 5, 6, 8, 9

• SUPER_MAIN_REGULATION_2ND = 19 or 20

• The following valve types are set on SOLELY CAREL VALVE driver:

• SUPER_EEV_TYPE ≠ 1 or 19

• SUPER_EEV_TYPE_2ND ≠ 1 or 19

• regulation is set on a UNIVERSAL driver:

• (SUPER_MAIN_REGULATION = 19)

If one of the previous conditions is met, data could bounce back between pCO and EVD evolution or communications could jam on the bus due to continuous pCO write attempts based on the type of communication protocol selected for the network. Parameter send and receive is generally no longer guaranteed.

Work As Single mode

A TWIN driver can be set as SINGLE with EVD_EVO by setting pin WorkAsSingle_EVD to TRUE. If this function is enabled, the driver becomes a SINGLE, permitting, for example, the use of auxiliary regulations.

these settings are visually signaled on the driver by the two OPEN and CLOSE leds blinking for the second valve.

if the connected driver is SINGLE, setting pin WorkAsSingle_EVD to TRUE has no effect.

Frequently changing parameters

EVD_EVO FB provides bitfield type VAR_INPUT VlvA_EVD.En_FrequentChgParams and VlvB_EVD.En_FrequentChgParams (see table EVD_EVO: outputs) to disable saving parameters in T memory and thus allowing the following to be frequently changed:

• SH_SET

• PID_Kp

• PID_Ti

• PID_Td

• LOP_Threshold

• LOP_Ti

• MOP_Threshold

• MOP_Ti

• LOW_SH

• LOW_SH_Ti

EVD_EVO programs driver EVD evolution to save parameters in its buffer memory (EEPROM) every two hours. This is to protect the device's EEPROM memory.

Setting the coolant gas

Set the type of coolant gas used with input pin RfrgVlv_A_EVD and RfrgVlv_B_EVD (the latter is only for twin drivers).

Support Gas Types

Gas numbering was offset by one position forward for driver firmware (i.e.: 1:R22...19: R407A).

Probe type values

Set the probe type using variables PrbCfg_EVD.S1.PrbTyp/PrbCfg_EVD.S2.PrbTyp/PrbCfg_EVD.S3.PrbTyp/PrbCfg_EVD.S4.PrbTyp. The Function Block automatically encrypts/decrypts to the driver.
