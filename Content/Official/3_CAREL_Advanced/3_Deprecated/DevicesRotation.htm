General operations
 

This function block is used to perform circuit and device rotation. The internal logic is illustrated below:

Rotation types

This function is able to manage 4 types of rotation:

  • FIFO (First in first out): the first device turned on will be the first turned off;
  • LIFO (Last in first out): the last device turned on will be the first turned off. It is also FOFI (First out first in) when all compressors have been on at least once;
  • Custom: fixed sequence based on a list entered by the user that sets the start order and a list that sets the stop order. The different sized device function can only be enabled with this type of management.
  • Timed: device on/off request is based on device operating hours. Specifically, the device that has totalled the lowest number of operating hours is selected to be started while the device with the highest number of operating hours is selected to be stopped. Operating hours are sent to this module via the two input pins TWork_High and TWork_Low in list format, structured as below:

Considering the limit of 999 for each position, the most significant and least significant parts must be separated using ABS and DIV functions.

Example

Second circuit first device operating hours.

Operating_Hours=1423

TWork_Low(4)=ABS(1000-Operating_Hours)

TWork_High(4)=DIV(Operating_Hours/1000)

Input setting: type of device

Inputs related to the device are the following:

  • Device size
    • Equal power: if all rotated devices have the same capacity
    • Different powers: populate DevPwr
  • Device type
    • Non-modulating
    • Modulating: set En_Inverter=1 and Comp_Power_Min; set Rotation_Type as Custom
  • Device load steps

    • CfgStage must be populated with the number of load steps for each device (see next paragraph)

Load steps

The following configurations are managed by the device rotation module:

 

CpppCppp, the criteria used to manage this configuration in the various rotation types are listed below:

  • FIFO
    • a) if the i-th device has n load steps with the highest ON priority at a given time, each of the following n ON requests will be assigned to the i-th device;
    b) if the i-th device has n load steps with the highest OFF priority at a given time, each of the following n OFF requests will be assigned to the i-th compressor;
    c) Points a) and b) are simultaneous (meaning that for alternating ON and OFF requests, ON will always be on the first-off or last-on compressor. OFF will always be on the first-on compressor)
  • LIFO
    • a) The i-th device can only be started if it is the first or if i-1 compressors are at 100%;
    b) The j-th load step can only be turned on for the compressor if load step j-1 is on;
    c) The i-th compressor can only be turned of if all its load steps are off;
    d) The j-th load step can only be turned off if j+1 is off and its compressor is at 100% and compressor i+1 is off.
  • Timed
    • a) if there are n load steps in the i-th device with the highest ON priority (where the highest priority is given by the device with the lowest operating hours) at a given time, each of the following n ON requests will be assigned to the i-th device;
    b) if there are n load steps in the i-th device with the highest OFF priority (where the highest priority is given by the device with the highest operating hours) at a given time, each of the following n OFF requests will be assigned to the i-th compressor;
    c) Points a) and b) are simultaneous (meaning that for alternating ON and OFF requests, ON will always be on the compressor with the lowest operating hours or last on. OFF will always be on the compressor with the highest operating hours)
  • Custom (on and off sequence set by the user)
    • a) if there are n load steps in the i-th device with the highest ON priority (where device priority is given by the start sequence set by the user) at a given time, each of the following n ON requests will be assigned to the i-th device;
    b) if there are n load steps in the i-th device with the highest OFF priority (where device priority is given by the start sequence set by the user) at a given time, each of the following n OFF requests will be assigned to the i-th compressor;
    c) Points a) and b) are simultaneous (meaning that for alternating ON and OFF requests, ON will always be on the compressor with the highest priority in the start sequence and OFF will always be on the compressor with the highest priority in the stop sequence.)

CCpppppp, the criteria used to manage this configuration in the various rotation types are listed below:

  • FIFO
    • a) The i-th load step can only start if all i-1 are on;
    b) The i-th load step can only stop if all i+1 are off;
    c) Start and stop order for load steps with the same i rating is FIFO, meaning the first off is turned on and first on is turned off;
  • LIFO
    • a) i-th load steps can only be turned on if all i-1 are on;
    b) i-th load steps can only be turned off if all i+1 are off;
    c) Load steps with the same rating are selected with LIFO logic: the one on the first on compressor is turned off first (and turned off after);
  • Timed
    • a) The i-th load step can only start if all i-1 are on;
    b) The i-th load step can only stop if all i+1 are off;
    c) Start and stop order for load steps with the same ratings follow the device start order meaning the one on the device with the lowest operating hours turns on and the one on the device with the highest operating hours turns off;
  • Custom (on and off sequence set by the user)
    • a) The i-th load step can only start if all i-1 are on;
    b) The i-th load step can only stop if all i+1 are off;
    c) Start and stop order or loads with the same rating follows device start order meaning the one on the device with the highest priority in the start sequence turns on and the one on the device with the highest priority in the stop sequence turns off;
Timed rotation management

DevicesRotation behaves in different ways depending on whether all the devices (except inverter-controlled devices, if present) have the same capacity (DevPwr not set) of they are different-sized devices.

  • Devices with the same size: The set rotation is followed. Thresholds are calculated by dividing the maximum request (1000) by the number of set devices. The activation differential for each device will be the same for all.
  • Devices with different sizes: Rotation can only be Custom. Threshold calculation depends on the size of each single device. This is dynamically calculated, meaning that thresholds are re-calculated whenever the number of available devices changes (in the event of alarms, forced off, capacity limits). Pump down request

DevicesRotation does not manage all the operations required to run pump-down in the circuit. According to the set pump-down type (PmpDwnTyp) it can set which compressor must run the procedure. Three types of pump-down can be set:

  • Pump-down at start: when the first circuit compressor on request is received, the module sets output DevRot[X]=5, where X (index) is the single device. Pump-down management starts after this (external to the module). After this, the function block that manages the single device returns value 14 in DevAvbl_Array. This way the rotation module freezes its status until the end of pump-down, which occurs as soon as DevAvbl_Array changes.
  • Pump-down at stop: when the last circuit compressor off request is received, the module sets output DevRot[X]=6, where X (index) is the single device. Pump-down is subsequently managed like at start.
  • Pump-down at both start and stop: the combination of the two types described above.
Circuit prevent: PreventCirc

DevicesRotation does not manage prevention control but can reduce capacity in a certain circuit following a prevent situation signal. This means that a changed PreventCirc input list element corresponds to the reduction in load steps in that circuit. The limit condition is represented by a single step on for that circuit. Input pin formatting to manage circuit prevent is illustrated below.

List index 0 1 2 3 4 5 6
Circuit 1 2 3 4 1 and 2 3 and 4 All circuits

As shown in the table above, the list is made up of 7 elements:

  • Index 0...3: represent the four circuits where a step is removed from the circuit at each value change based on the selected rotation.
  • Index 4: a step is removed from either circuit 1 or 2 when the value changes based on the selected rotation.
  • Index 5: a step is removed from either circuit 3 or 4 when the value changes based on the selected rotation.
  • Index 6: represents the entire unit, where a step is removed from any one of the unit circuits at each change based on the selected rotation

In all cases, no other step will be added to the circuit in prevent when request increases while steps can always be removed if the request decreases.

prevent conditions are reset for the circuit or unit when the module reads 0 in the relevant list element.

Forcing circuit device on

All devices in a certain circuit can be forced on. To do this, switch input FrcOnDevCirc to 1 where X is the required circuit. Devices are forced on according to load-up time. After forcing, circuit devices will be turned on/off according to the normal request.

This function is usually used to force all circuit compressors on during defrosting. When all devices in a circuit are forced on, the thermostat request moves on to the next circuit.

Forced rotation

This function forces device rotation in a single circuit. A device is forced off and the consequent start of another is always in the same circuit. Devices are selected according to the set rotation type. Devices are turned on and off at the same time even with load step devices.

DevicesRotation runs forced rotation when the relevant input pin switches from 0 to 1. If no device can be turned on, forcing won’t run.

Next device ON and next device OFF

Through output pin DevNextOn e CircNextOn, this device makes the next device available to be turned on and, through DevNextOff e CircNextOff , the next to be turned off. The choice is made according to the set type of rotation and compressor safety times, sent to Mod_Devices_Rotation_2 by a list CntDwnSafT_Array, populated with minimum shutdown times. At the end of this time, the compressor can be turned back on (in safety conditions) and the CntDwnFrcOnT list populated with minimum start time. Only at the end of this time can the compressor be turned off (unless an alarm is triggered).

Synchronisation between two modules

More than one DevicesRotation can be used in the same application to manage several rotation groups like, for example, in cooling units where two separate compressor lines can be managed.

A maximum of two rotation groups can interact to have them work in synergy. The goals are:

  • • To avoid starting several devices at the same time. For this reason, DevicesRotation provides one output pin, Stop_Other_Device, and one input pin, WaitOtherDev, which must be linked. In detail:

    • if input WaitOtherDev is 1, the module doesn’t run any operation until it returns to 0;

    • output pin StopOtherDev switches to 1 at each device on request. It returns to 0 as soon as the device is on based on device feedback (DevAvbl_Array[X]=2).
  • • Forcing on at minimum step. For this reason, DevicesRotation provides input pin Cascade_Optimization which, when 1, forces the request to minimum step threshold if no device request is in progress, thus requesting its start.
