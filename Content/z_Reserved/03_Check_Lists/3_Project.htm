Project - Check list

Questa sezione serve per fare la check-list prima di pubblicare l'help ufficiale.

Rilasci intermedi
  • Blocchi nuovi, controllare presenza segnaposto release notes per link da KSA a tabella R.N.
  • Controllare con Show Log tutti i commit che ci sono stati e cosa sono andati a modificare.
  • Nel caso di bugfix nel Branch, fare marge.

 

Pre- inserimento in c.strategy help

Targhet HTML5

  • Conditions doppie, triple, etc generano inserimento nell'output anche se solo 1 di queste è vera. Verificare la correttezza dei contenuti in output
  • Pagina introduttiva al c.library controllare le argomentazioni se sono ok oppure da aggiornare
  • Controllare che in compilazione restituisca 0 errori, 0 waring.
  • Panel Project organizer -> Reports - Generare reports a doc per la checklist e verificarli:
    • Bookmarks.flrep
    • Conditions_Report.flrep
    • NotInToc
    • Nwords
    • Unused_Images.flrep
    • ...
  • Menù View -> Project analysis controllare cosa restituisce:
    • Broken bookmarks
    • Broken links
    • File with changes
    • File with annotations
    • Topics not in index
    • Topics not in selected tocs.
  • Oppure con Madcup Analyzer
  • Controllare il Tocs "c.Library_non-print" siano allineati al Content Explorer.
  • Controllare la presenza di link che puntano a versioni assolute di URL e che devo essere aggiornate (es. How_to_create_library.htm)
  • Targhet c.suite_setup -> fare tutte le prove in locale e non sul server intranet.
  • Targhet c.suitedocs -> fare tutte le prove in ambiente di pre-produzione.
  • View -> File list controllare tutti i Topic, le Conditions, Posizione se ok. Aggiungere la proprietà CAREL INDUSTRIES SpA, lingua EN-US se manca.
  • Controllare la presenza dei marchi © ® ™ dove è necessario
  • Controllare versionamento e data del progetto nei menù dropdown a piè di pagina.
  • Controllare lista dei blocchi ufficiali in KSA e conditions che nascondono i contenuti in beta version.
  • Controllare cartelle images Content Explorer con le relative conditions (es. ppt).
  • Controllare cartelle Outputs via file system (es. se sono rimasti topics old, img non in cartelle predisposte, images riservate, ecc).
  • Controllare "Links" e "Cross references" che siano aggiornati e funzionanti (es. topic Gas types supported)
  • c.suite version e OS da controllare per tutte le librerie di blocchi se allineati all'ultima versione (se necessario).
  • Controllare la presenza del p.warning per blocchi deprecati.
  • Controllare che i blocchi abbiano aggiornato la icona di blocchetto nel menù di output.
  • Project Organizer -> Conditional Text -> tasto dx -> View links vedere se realmente sono corrette in giro per i vari Topics.
  • Notepad ++ controllare, menu cerca, trova nei file... per tag che non ci devono essere.
  • Test con Search da output (es. riferimenti a uPC, uPC3, OEM+ hws riservati che non deve essere presenti).
  • Aggiornare la pagina di statistiche "Number_of_blocks" con il numero dei blocchi presenti
  • Controllare che i nuovi blocchi inseriti abbiano la demo
  • Fare una prova con più browser (EI, Firefox, Crome, ...)
  • L'anno del Copyright, in piè di pagina, da tenere aggiornato con l'anno corrente.
  • Fare commit della versione su SVN:
    • https://svncarel.carel.com/svn/csuite_help/trunk/c.library_ENG
  • Fare backup dei binari su:
    • \\Backup\Varie\csuite_help_binaries\
  • Fare pubblicazione su ftp Intranet
  • FARE branch di versione in SVN
